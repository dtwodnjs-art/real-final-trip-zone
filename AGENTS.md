# TripZone Working Context

## Project Priority

- 이 프로젝트의 기준은 끝까지 설계도다.
- 참고 UI는 `trip-demo`, Airbnb, 여기어때의 좋은 부분만 선별 반영한다.
- 설계도에 없는 기능은 임의로 추가하지 않는다.
- 설계도에 있는 흐름과 역할은 삭제하지 않는다.

## Role Boundary

- 이 저장소에서 내 역할은 프론트엔드 구현 우선이다.
- 백엔드는 팀원이 이어받을 수 있도록 최소 골격과 가이드만 유지한다.
- 백엔드 구현을 확장하지 않는다. 필요한 경우에도 루트, 패키지, README, 최소 연결점까지만 다룬다.

## UI Direction

- 목표는 팀원에게 넘겨도 되는 실서비스급 여행 예약 프론트다.
- 메인은 여행 사이트 메인으로 유지한다.
- 문서 허브, 제출물 허브, 설계 설명 느낌을 사용자 메인에서 전면에 내세우지 않는다.
- 상세, 리스트, 예약, 마이페이지, 판매자/관리자 화면은 설계도 범위 안에서 고도화한다.
- 디자인 기준은 `frontend-skill` 원칙을 따른다.

## Frontend Skill Rules To Keep

- 카드보다 구성과 위계를 먼저 본다.
- 이미지가 실제 역할을 해야 한다.
- 색 포인트는 의미 있는 곳에만 쓴다.
- 카피는 줄이고, 상태/가격/후기/행동을 먼저 보이게 한다.
- 화면이 밋밋하다고 박스와 그림자를 먼저 늘리지 않는다.
- spacing, alignment, scale, contrast로 먼저 해결한다.
- 제품 UI에서는 마케팅 카피보다 유틸리티 카피를 우선한다.

## Current Fixed Intent

- 메인페이지는 현재 방향을 유지한다.
- 상세페이지는 Airbnb + 여기어때의 좋은 부분을 참고하되 그대로 복제하지 않는다.
- 리스트/상세/예약/마이페이지는 하나씩 끝내고 다음으로 넘어간다.
- 한 페이지를 동시에 여러 방향으로 흔들지 않는다.

## Working Method

- 구현 전 현재 상태와 영향 범위를 먼저 읽는다.
- 한 번에 한 페이지씩 집중해서 끝낸다.
- 큰 리디자인보다 현재 구조에서 부족한 부분만 정확히 보강한다.
- 사용자가 직접 짚은 문제를 최우선으로 수정한다.

## Verification

- 성공이라고 말하기 전에 빌드나 실제 증거를 먼저 확인한다.
- 프론트 변경 후 기본 검증은 `npm run build`다.
- 검증하지 못한 내용은 `UNVERIFIED`로 둔다.
