# TripZone 프론트 연동 가이드

## 1. 문서 목적

이 문서는 TripZone 프론트 화면을 실제 API와 연결할 때 필요한 응답 기준을 정리한 가이드다.

현재 프론트는 설계도와 mock 데이터를 기준으로 화면을 구성하고 있다.
따라서 Oracle 연결이 먼저 성공해도 조회 필드가 최소 수준이면 화면은 단출하게 보일 수 있다.

핵심은 다음과 같다.

- UI 레이아웃이 깨진 것이 아니라 데이터가 덜 채워진 상태일 수 있다.
- 백엔드는 설계 문서 기준 도메인을 유지하되, 프론트가 실제로 그리는 데 필요한 응답 shape를 맞춰야 한다.
- 이 문서는 프론트와 백엔드가 맞춰야 하는 최소 응답 계약을 정의한다.

## 2. 기본 원칙

### 2.1 설계도 우선

- 도메인명, 상태값, 기본키 명명은 기존 설계 문서를 따른다.
- 프론트 연결 시에도 `com.kh.trip`, `InquiryRoom`, `InquiryMessage` 기준을 유지한다.
- 팀원 백엔드 구현이 설계도와 충돌하면 설계도 기준을 먼저 유지하고, 충돌 사실은 별도 메모로 관리한다.

### 2.1.1 현재 백엔드 구현과 충돌하는 항목

2026-03-31 기준으로 확인된 충돌 항목:

- 문의 모델:
  - 설계도: `InquiryRoom`, `InquiryMessage`
  - 현재 백엔드: 단일 `Inquiry`
- 문의 상태:
  - 설계도: `OPEN`, `ANSWERED`, `CLOSED`, `BLOCKED`
  - 현재 백엔드: `PENDING`, `COMPLETED`, `DELETE`
- 문의 타입:
  - 설계도/프론트: 숙소/예약/결제/시스템 문의 흐름
  - 현재 백엔드: `SYSTEM`, `MANAGEMENT`, `ETC`
- 인증 로그인 요청:
  - 프론트 입력 UI: `email`, `password`
  - 현재 백엔드 요청 필드: `loginId`, `password`

판정:

- 문의 구조는 설계 문서의 `InquiryRoom`, `InquiryMessage` 기준과 아직 어긋나 있다.
- 판매자 예약 상태 변경은 `PATCH /api/seller/bookings/{bookingNo}/status`로 1차 연결됐다.
- 마이페이지 문의 목록은 현재 `GET /api/mypage/inquiries` 기준으로 연결됐다.
- 인증 필드명은 프론트에서 `loginId`로 어댑터 처리 중이다.

### 빠른 판정표

| 항목 | 설계도 기준 유지 여부 | 처리 방식 |
|---|---|---|
| 숙소 | 유지 | 현재 백엔드 DTO를 `services`에서 어댑터 처리 |
| 예약 | 유지 | 현재 백엔드 DTO를 `services`에서 어댑터 처리 |
| 결제 | 유지 | 현재 백엔드 DTO를 `services`에서 어댑터 처리 |
| 리뷰 | 유지 | 현재 백엔드 API를 프론트에서 연결 |
| 인증 | 합의 필요 | `email` 또는 `loginId` 중 하나로 통일 |
| 문의 | 강제 유지 | 목록은 `GET /api/mypage/inquiries` 사용, 상세/메시지 구조는 추가 정리 필요 |
| 관리자 운영 | 강제 유지 | 조회 일부만 연결, 상태 변경 API는 미완료 |

짧게 정리하면:

- `숙소/예약/결제/리뷰`: 프론트 어댑터로 해결
- `문의`: 목록은 연결됨, 상세 메시지 구조는 백엔드 추가 정리 필요
- `관리자 운영`: 조회 위주 연결, 액션 API는 추가 구현 필요
- `인증`: `loginId` 어댑터는 완료, 소셜 로그인은 공급자 설정까지 확인 필요

### 2.2 화면 단위 응답

- 백엔드 엔티티를 그대로 노출하지 않는다.
- 목록 화면은 화면에 필요한 조합형 응답을 내려주는 것이 안전하다.
- count, summary, 상태 라벨, 이동용 id는 프론트에서 다시 계산하기보다 API에서 함께 주는 편이 안정적이다.

### 2.3 최소 응답 + 확장 응답 분리

- 초기에 필요한 최소 필드를 먼저 맞춘다.
- 이후 badge, summary, count, 최근 상태 같은 보조 필드를 확장한다.

## 3. 현재 프론트 구조

현재 프론트는 `page -> feature -> data` 기준에서 시작했고, 현재 연결 경계는 `page -> service -> data/mockDb` 쪽으로 정리돼 있다.

- `pages/`
  - 라우트 진입점
  - 화면 조립과 최소 상태만 담당
- `features/`
  - 화면별 UI 블록
  - viewModel
  - 유틸
- `services/`
  - mock 또는 실제 API 선택 경계
- `lib/`
  - `appClient.js`: 데이터 소스 기준
  - `mockDb.js`: localStorage 기반 mock DB
- `data/`
  - mock 데이터와 shape 기준

백엔드 연결 시 가장 먼저 볼 위치는 아래다.

### 3.1 공통 진입

- 인증 세션: `frontend/src/features/auth/authSession.js`
- 인증 viewModel: `frontend/src/features/auth/authViewModels.js`
- `frontend/src/data/siteData.js`는 더 이상 메인 연결 지점이 아니다.
- 화면 연결은 각 도메인 data/viewModel 파일 기준으로 본다.

### 3.2 도메인별 연결 지점

#### 메인 / 리스트 / 상세 / 예약

- 홈:
  - page: `frontend/src/pages/common/HomePage.jsx`

- 숙소 리스트:

- 예약/결제:

  - feature: `frontend/src/features/booking/`

#### 마이페이지

### 3.3 연결 방식

권장 방식은 아래 순서다.

1. `lib/appClient.js`에서 데이터 소스 기준을 잡는다.
2. `services/*.js`에서 mock 또는 API를 선택한다.
3. `data/*.js`의 mock shape를 실제 API 응답 shape와 맞춘다.
4. 필요한 가공은 `features/*ViewModel.js`에서 수행한다.
5. `pages/`는 가능하면 수정하지 않는다.

즉, API가 붙을 때 가장 먼저 수정할 파일은 보통 `services`다.

### 3.4 2026-03-31 기준 현재 연결 상태

- 숙소 목록/상세, 객실, 리뷰 기본 조회는 실제 API 기준으로 연결됐다.
- 마이페이지 예약/결제/쿠폰/마일리지/프로필 조회는 실제 API 기준으로 연결됐다.
- 마이페이지 문의 목록은 `GET /api/mypage/inquiries` 기준으로 연결됐다.
- 문의 상세는 아직 `GET /api/inquiry/{id}` + `GET /api/comment/list` 조합을 사용한다.
- 판매자 예약 상태 변경은 `PATCH /api/seller/bookings/{bookingNo}/status` 기준으로 연결됐다.
- 판매자 숙소/객실 상태 변경은 기존 숙소/객실 API를 사용해 연결됐다.
- 관리자 조회 화면은 일부 연결됐지만 상태 변경 액션은 아직 미연결 구간이 남아 있다.
- 카카오 로그인 실패의 직접 원인은 프론트 환경변수 누락이었고, `.env.local` 기준 설정 후 인가 페이지 진입까지 확인됐다.
- 네이버 로그인은 공급자 설정 오류, 구글 로그인은 테스트 브라우저 세션 조건 문제로 추가 검증이 필요하다.

## 4. 공통 응답 계약

목록성 화면은 아래 공통 필드를 우선 고려한다.

- `id`: 화면 항목의 고유 식별자
- `status`: 실제 상태값
- `statusLabel`: 화면 노출용 상태 라벨
- `createdAt` 또는 `regDate`: 정렬 기준 시각
- `updatedAt`: 최근 변경 시각
- `actions`: 이동에 필요한 참조 id 묶음
- `summary`: 상단 요약 영역에 필요한 count, amount, balance

백엔드 기존 DTO가 이 계약과 다르면 `services`에서 어댑터를 둔다.
현재 팀원 백엔드 코드는 `lodgingNo`, `bookingNo`, `roomNo`, `userCouponNo`, `paymentNo`를 주로 사용한다.

## 4.1 소셜 로그인 환경변수

프론트 소셜 로그인 버튼은 아래 환경변수가 없으면 동작하지 않는다.

- `VITE_GOOGLE_CLIENT_ID`
- `VITE_KAKAO_CLIENT_ID`
- `VITE_NAVER_CLIENT_ID`

현재 프론트는 `frontend/src/features/auth/authViewModels.js`에서 위 값을 직접 읽고, 값이 비어 있으면 즉시 에러를 던진다.
로컬 실행 시에는 `frontend/.env.local` 또는 동등한 Vite 환경파일에 반드시 선언해야 한다.

예시:

```json
{
  "summary": {
    "totalCount": 3,
    "activeCount": 1,
    "completedCount": 2
  },
  "items": [
    {
      "id": 301,
      "status": "CONFIRMED",
      "statusLabel": "예약 확정",
      "createdAt": "2026-03-25T09:00:00",
      "updatedAt": "2026-03-25T10:15:00",
      "actions": {
        "bookingId": 301,
        "lodgingId": 12,
        "roomId": 101
      }
    }
  ]
}
```

## 6. 프론트 연결 우선순위

1. 내 예약
2. 쿠폰
3. 마일리지
4. 결제 내역
5. 문의센터

현재 파일 구조 기준 우선 연결 지점은 아래와 같다.

1. `frontend/src/data/mypageData.js`
2. `frontend/src/features/mypage/mypageViewModels.js`
3. `frontend/src/data/bookingData.js`
4. `frontend/src/features/booking/bookingViewModel.js`
5. `frontend/src/data/lodgingData.js`
6. `frontend/src/features/dashboard/dashboardViewModels.js`

이 순서가 맞는 이유:

- 마이페이지에서 사용 빈도가 높다.
- summary와 count가 빠지면 화면 체감이 크게 단순해진다.
- join 필드가 필요한 화면이어서 초기에 계약을 맞춰두는 편이 좋다.

## 7. 프론트 fallback 원칙

- summary가 없으면 0으로 처리
- `statusLabel`이 없으면 프론트 기본 매핑 사용
- 썸네일이 없으면 기본 이미지 사용
- count, label, preview가 없더라도 레이아웃이 깨지지 않게 방어
