# TripZone 설계도-백엔드 충돌 정리

이 문서는 설계 문서와 현재 팀원 백엔드 구현 사이에서 충돌이 있는 지점을 빠르게 판단하기 위한 문서다.

## 판단 규칙

- 설계 문서와 현재 백엔드 코드가 같으면: 그대로 연결
- 설계 문서와 현재 백엔드 코드가 다르지만 `숙소/예약/결제/리뷰`면: 프론트 `services` 어댑터로 우선 해결 가능
- 설계 문서와 현재 백엔드 코드가 다르고 `문의/관리자 운영`이면: 백엔드가 설계도 기준으로 따라와야 함
- 인증처럼 설계 문서가 세부 필드를 고정하지 않은 경우는: 먼저 합의

## 한눈에 정리

| 항목 | 설계도 기준 | 현재 백엔드 상태 | 판단 |
|---|---|---|---|
| 숙소 | 설계도 유지 | DTO 필드명 차이 가능 | 서비스 어댑터로 해결 가능 |
| 예약 | 설계도 유지 | `bookingNo` 등 명명 차이 가능 | 서비스 어댑터로 해결 가능 |
| 결제 | 설계도 유지 | `paymentNo/paymentId` 구분 가능 | 서비스 어댑터로 해결 가능 |
| 리뷰 | 설계도 유지 | API 연결 가능 | 연결 우선 |
| 인증 | 문서 해석 가능 | `email` vs `loginId` | 합의 필요 |
| 문의 | `InquiryRoom`, `InquiryMessage` | 단일 `Inquiry`일 수 있음 | 백엔드 수정 필요 |
| 관리자 운영 | admin API 필요 | 구현 미완성 가능 | 백엔드 구현 필요 |

## 충돌 상세

### 인증

- 프론트 mock: `email`
- 일부 현재 백엔드: `loginId`

판정:

- `합의 필요`

### 문의

설계도 기준:

- `InquiryRoom`
- `InquiryMessage`
- 상태: `OPEN`, `ANSWERED`, `CLOSED`, `BLOCKED`

현재 백엔드가 아래라면 충돌이다.

- 단일 `Inquiry`
- 상태: `PENDING`, `COMPLETED`, `DELETE`
- 타입: `SYSTEM`, `MANAGEMENT`, `ETC`

판정:

- `백엔드가 설계도 최신본을 아직 미반영`

### 관리자 운영

설계도 기준:

- 판매자 승인 관리
- 회원 상태 관리
- 이벤트/쿠폰 운영
- 문의 모니터링
- 감사 로그

현재 admin API가 비어 있거나 일부만 있으면 충돌이다.

판정:

- `백엔드 구현 필요`

## 팀원에게 바로 전달할 문장

- `숙소/예약/결제/리뷰는 지금 백엔드 DTO로 붙여도 되고, 프론트 service에서 키만 맞추면 됩니다.`
- `문의는 설계도 최신본이 InquiryRoom/InquiryMessage 구조라서 현재 백엔드 단일 Inquiry 모델은 미반영 상태입니다.`
- `관리자 운영 API는 설계도에는 있지만 현재 백엔드 구현이 비어 있는 영역입니다.`
- `인증은 email/loginId 중 하나로 먼저 합의하고 고정해야 합니다.`
