# TripZone 이미지 경로 / 시드 데이터 가이드

이 문서는 현재 mock 이미지가 실제 DB/API 연결 후에도 최대한 그대로 보이게 하는 기준을 정리한 문서다.

## 결론

초기 연결 단계에서는 이미지를 DB에 파일 자체로 넣지 말고 `경로 문자열`로 저장하는 방식이 맞다.

즉:

- mock 이미지 경로를 seed 기준으로 사용
- DB에는 `image_url` 또는 `image_path` 문자열 저장
- API는 그 값을 그대로 반환
- 프론트는 기존 렌더 로직 유지

## 권장 흐름

`mock image path -> seed data -> DB column -> API response -> frontend render`

## 왜 이렇게 가는지

- 프론트 수정이 최소다
- mock과 실제 화면 차이가 작다
- 초기 연결이 빠르다
- 나중에 업로드 시스템을 붙여도 API 필드명만 유지하면 된다

## 현재 기준

현재 프론트는 mock 이미지 경로를 그대로 렌더하는 구조가 많다.

따라서 초기 백엔드/DB 연결 단계에서는 아래 원칙을 유지한다.

- 파일 저장소 구현보다 `경로 유지`를 먼저 한다
- BLOB 저장보다 `문자열 URL/path 저장`을 먼저 한다

## 컬럼 예시

### 숙소 이미지

- `LODGING_IMAGES.IMAGE_URL`
- `LODGING_IMAGES.SORT_ORDER`
- `LODGING_IMAGES.IS_PRIMARY`

### 리뷰 이미지

- `REVIEW_IMAGES.IMAGE_URL`

### 이벤트/배너 이미지

- `EVENTS.BANNER_IMAGE_URL`

## 응답 예시

```json
{
  "lodgingId": 12,
  "lodgingName": "서프코브 풀빌라",
  "thumbnailUrl": "/images/lodging/surf-cove/thumb.jpg",
  "galleryImages": [
    "/images/lodging/surf-cove/01.jpg",
    "/images/lodging/surf-cove/02.jpg"
  ]
}
```

## 정적 파일 기준

초기 기준은 현재 프론트 정적 자산 경로 체계를 유지하는 방식으로 본다.

예:

- `frontend/public/images/...`

중요:

- 백엔드가 이미지를 직접 서빙하든
- 프론트 정적 자산으로 유지하든

둘 중 하나를 택하더라도 API가 반환하는 최종 `imageUrl` 문자열은 안정적으로 유지해야 한다.

## 나중에 업로드 시스템을 붙일 때

업로드 시스템은 나중에 붙여도 된다.

그때 바뀌는 것은:

- 실제 파일 저장 위치
- 업로드 처리
- CDN/S3/스토리지 URL

바뀌지 않게 유지해야 하는 것은:

- 프론트가 받는 `imageUrl`
- 썸네일/갤러리 이미지 배열 구조

즉, 지금은 `경로 문자열 유지`가 우선이다.
