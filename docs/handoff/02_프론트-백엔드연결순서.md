# TripZone 프론트-백엔드 연결 순서

이 문서는 백엔드 팀원이 프론트에 실제 API를 붙일 때 어디부터 보면 되는지 정리한 문서다.

## 핵심 원칙

- `pages`를 먼저 뜯지 않는다.
- `features`를 먼저 뜯지 않는다.
- 먼저 `services`와 `appClient`를 교체한다.

현재 프론트 구조:

- `frontend/src/lib/appClient.js`
- `frontend/src/lib/mockDb.js`
- `frontend/src/services/*.js`
- `frontend/src/data/*.js`
- `frontend/src/features/*`
- `frontend/src/pages/*`

현재 데이터 흐름:

`page -> feature/viewModel -> service -> appClient -> mockDb/data`

실제 API 연결 후 목표 흐름:

`page -> feature/viewModel -> service -> appClient -> HTTP API`

## 실제 교체 순서

1. `frontend/src/lib/appClient.js`
2. `frontend/src/services/lodgingService.js`
3. `frontend/src/services/bookingService.js`
4. `frontend/src/services/mypageService.js`
5. `frontend/src/services/dashboardService.js`
6. 필요한 경우만 `frontend/src/features/*` 보정

## 연결 난이도 판단

- 예약
- 결제

### 구조 수정이 필요한 영역

- 인증 `email` vs `loginId`

## 응답 shape 원칙

백엔드 DTO가 현재 아래 키를 써도 괜찮다.

- `lodgingNo`
- `bookingNo`
- `roomNo`
- `paymentNo`

다만 프론트 서비스에서 최종 화면 shape는 아래처럼 맞춘다.

```json
{
  "bookingId": 301,
  "lodgingId": 12,
  "roomId": 101,
  "bookingStatus": "CONFIRMED",
  "bookingStatusLabel": "예약 확정",
  "totalPrice": 189000
}
```

즉, 엔티티/기존 DTO 명명과 화면 응답 명명은 서비스 어댑터에서 분리해도 된다.

## 작업 중 체크리스트

- 설계도 상태값을 유지하는가
- `services`를 기준으로 API를 연결하는가
- 목록 응답에 `id`, `status`, `statusLabel`, `summary`가 필요한지 확인했는가
- `pages`를 직접 수정하기 전에 `services`로 해결 가능한지 먼저 봤는가
